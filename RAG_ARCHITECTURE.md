# 统计讲义智答 RAG 搭建过程与原理解析

本文档面向两个目标：

- 让你能从工程角度复现这套中文统计学 RAG
- 让你能在课堂展示时讲清楚它为什么这样设计

这份说明是基于当前仓库里的实际实现整理出来的，不是泛泛而谈的 RAG 教程。

## 1. 项目目标

本项目是一个面向应用统计课堂展示的中文 RAG 系统，知识库来源于清华大学公开统计讲义 PDF，用户可以提出自然语言问题，系统会：

1. 先在讲义中检索相关内容
2. 对候选内容做重排
3. 将最相关的证据交给回答模型
4. 生成带来源依据的中文解释
5. 在网页端把 Markdown 和公式渲染出来，便于展示

它解决的问题不是“让模型背统计学”，而是“让模型基于统计学讲义回答问题，并尽量减少脱离资料的胡乱发挥”。

## 2. 为什么选 RAG

对于课堂展示型项目，RAG 比纯聊天模型或微调更合适，原因有三点：

- 知识可控：回答主要依赖本地讲义，而不是完全依赖大模型参数记忆
- 更新方便：只要替换 PDF 并重建向量库，不需要重新训练
- 可解释性更强：可以展示引用页码和相关片段，便于课堂答辩

所以这个项目的核心思想是：

`用户问题 -> 检索知识 -> 重排证据 -> 约束生成 -> 展示答案`

## 3. 本项目的整体架构

可以把它理解成下面这条链路：

```text
统计讲义 PDF
  -> 文本抽取
  -> 清洗页眉页脚和页码
  -> 按页聚合切分 chunk
  -> Embedding 向量化
  -> Chroma 向量库存储
  -> 用户提问
  -> Query Embedding
  -> 向量召回 Top-K
  -> Rerank 重排
  -> 将最相关 chunk 拼成上下文
  -> 交给回答模型生成答案
  -> Web 端渲染 Markdown 与公式
```

这套流程主要由 [stats_rag.py](stats_rag.py) 和 [stats_rag_web.py](stats_rag_web.py) 两部分组成：

- [stats_rag.py](stats_rag.py) 负责知识库构建、检索、重排、回答生成
- [stats_rag_web.py](stats_rag_web.py) 负责网页界面、API 接口、答案展示和公式渲染

## 4. 知识库为什么这样选

当前知识库使用的是清华大学公开统计讲义，放在 [knowledge_base/tsinghua_stats](knowledge_base/tsinghua_stats)。

这样选的原因很实际：

- 内容权威：来自公开课程讲义，适合作为课堂展示资料
- 内容集中：主题围绕统计学，避免知识范围过散
- 语言匹配：中文讲义更适合中文课堂展示
- 可引用：RAG 检索出的内容可以明确对应到某个 PDF 和页码

对于应用统计专业来说，这类资料特别适合做“概念解释型”和“方法比较型”问答，例如：

- 什么情况下使用方差分析
- 泊松回归适用于什么数据
- 广义线性模型与线性回归有什么区别
- 自助法为什么可以估计不确定性

## 5. 环境变量与模型分工

项目把三类模型拆开配置，而不是全部混用一个模型：

- Generation 模型：负责最终回答
- Embedding 模型：负责把文本编码成向量
- Rerank 模型：负责把召回结果重新排序

在当前项目中，默认使用的是：

- Generation: `Qwen/Qwen3-14B`
- Embedding: `Qwen/Qwen3-Embedding-4B`
- Rerank: `Qwen/Qwen3-Reranker-4B`

这种拆分符合 RAG 的常见工程思路，因为这三类任务的能力需求并不一样：

- 回答模型更关注语言表达、推理和组织答案
- Embedding 更关注语义空间中的相似性
- Rerank 更关注“候选片段是否真的和问题最匹配”

[stats_rag.py](stats_rag.py) 中的 `AppConfig` 会从 `.env` 读取这些参数，并允许为检索、切分、超时等行为设置默认值。

## 6. 搭建过程

这一节讲的是这套项目是怎么一步步搭起来的。

### 6.1 读取 PDF

项目使用 `pypdf` 读取 PDF 文本，核心函数是 [stats_rag.py](stats_rag.py) 里的 `extract_pdf_pages()`。

它的工作流程是：

1. 逐页读取 PDF
2. 提取每一页的文本
3. 按行拆分
4. 统计每一行在整份 PDF 中重复出现的频率

这一步的目的不仅是“读出文字”，更重要的是为后续清理噪声做准备。

### 6.2 清理页眉、页脚和页码

课堂讲义 PDF 往往有很多重复信息，比如：

- 页码
- 章节抬头
- 页眉页脚
- 很短但反复出现的提示文本

如果这些内容不清掉，会导致两个问题：

- 向量库里塞进很多无意义重复文本
- 检索时容易召回“每页都一样的页眉页脚”

所以项目在 [stats_rag.py](stats_rag.py) 里用了 `is_probable_boilerplate()` 来做启发式过滤。它会优先移除：

- 纯页码
- 过短且高频重复的行
- 在整份 PDF 中出现过多次的短行

这里不是完美 OCR，而是“课堂展示足够实用”的工程策略。

### 6.3 为什么按页聚合切分 chunk

RAG 不能直接把整本 PDF 一次性喂给模型，所以需要把讲义切成可检索的小片段，也就是 chunk。

本项目的切分逻辑在 [stats_rag.py](stats_rag.py) 的 `chunk_pages()` 中，思路不是机械地按固定 token 截断，而是：

- 以页面为基本语义单位
- 聚合若干页文本，直到接近目标长度
- 保留一定页重叠，减少上下文断裂

当前默认参数是：

- `CHUNK_TARGET_CHARS=550`
- `CHUNK_OVERLAP_PAGES=1`
- `MIN_CHUNK_CHARS=120`

这样设计的原因是：

- 讲义按页天然带有结构感
- 统计学概念通常会跨半页到数页展开
- 适度重叠可以提升召回连续概念的概率

如果 chunk 太小，会导致上下文碎片化；如果太大，则会降低召回精度并增加 rerank 和生成成本。

### 6.4 建立向量库

切好 chunk 后，项目会调用 Embedding 模型，把每个 chunk 编码成向量。这个过程由 [stats_rag.py](stats_rag.py) 中的 `SiliconFlowEmbedder` 完成。

向量化后的数据会写入 Chroma，本地存储路径默认是：

- `rag_store/stats_chroma`

同时还会写一个清单文件：

- `rag_store/stats_manifest.json`

这个清单记录了：

- 知识库目录
- PDF 数量
- chunk 数量
- embedding 模型
- rerank 模型
- generation 模型
- 当前切分参数

它的作用类似“实验配置快照”，便于你后续回溯当前索引是按什么参数构建出来的。

### 6.5 用户提问时如何检索

当用户输入问题时，系统不会直接把问题扔给回答模型，而是先走检索链路：

1. 用 Embedding 模型把问题编码成 query vector
2. 在 Chroma 里做向量相似度召回
3. 先拿到一批候选 chunk

这一步在 [stats_rag.py](stats_rag.py) 的 `search()` 中实现。

当前默认先召回 `RETRIEVAL_TOP_K=12` 个候选片段。

这里的原理可以简单理解为：

- 讲义片段和问题都被映射到同一个向量空间
- 语义越接近，向量距离通常越近
- 所以可以通过“找最近邻”先粗召回一批可能相关的内容

这一步速度快，但不一定最精确，所以还需要 rerank。

### 6.6 为什么还需要 rerank

只做向量检索有一个常见问题：召回结果“差不多相关”，但不一定“最适合回答这个问题”。

例如：

- 问题是“什么时候适合用方差分析”
- 召回结果可能同时包含“方差分析定义”“方差分解公式”“回归与方差分析关系”

这些都相关，但真正最该排在前面的，不一定是向量距离最近的那个。

所以项目在 [stats_rag.py](stats_rag.py) 中加入了 `SiliconFlowReranker`，对召回的候选 chunk 再排一次序。

当前默认只保留 `RERANK_TOP_N=5` 条结果。

它的作用可以理解成：

- Embedding 负责“先广泛找一批相关内容”
- Rerank 负责“从这批内容里挑出最像正确证据的几个”

这个两阶段检索，是本项目回答质量明显优于“纯向量搜索”的关键原因之一。

### 6.7 回答模型如何被约束

完成检索和重排后，项目会把最相关的几个 chunk 拼接成上下文，交给回答模型生成答案。

这部分由 [stats_rag.py](stats_rag.py) 中的 `Generator.answer()` 完成。

它会构造两个重要部分：

- `system_prompt`：告诉模型它是应用统计课程助教，要严格依据讲义回答
- `user_prompt`：把用户问题和检索到的资料一起发给模型

当前提示词的目标很明确：

- 严格依据检索到的讲义内容回答
- 优先给出清晰、适合课堂展示的中文解释
- 证据不足时明确说明
- 尽量说明概念、适用场景、关键假设和结果解释
- 如果出现公式，用 LaTeX 语法输出
- 在答案末尾附上引用资料编号

这一步体现的是 RAG 最关键的原则：

不是让大模型“自由发挥”，而是让大模型“围绕证据组织答案”。

## 7. Web 端是怎么接上 RAG 的

命令行版适合开发和测试，但课堂展示更适合网页界面，因此项目又实现了一个轻量 Web 服务 [stats_rag_web.py](stats_rag_web.py)。

它包含三层：

- 页面模板：负责展示输入框、答案区、来源区
- API 接口：接收问题并返回回答
- PDF 路由：允许直接打开原始 PDF

主要接口有：

- `/`：网页首页
- `/api/ask`：提问接口
- `/api/health`：健康检查
- `/kb/{name}`：打开知识库 PDF

这意味着网页端本质上只是 CLI RAG 的展示壳，真正的检索与生成逻辑仍然复用 [stats_rag.py](stats_rag.py)。

## 8. 为什么公式显示需要单独设计

统计学问答里很容易出现：

- 假设检验统计量
- 回归模型表达式
- 概率分布公式
- 方差分析分解式

如果网页端直接把模型输出当纯文本显示，用户会看到：

- Markdown 标题原样显示为 `###`
- 公式原样显示为 `$$ ... $$`

这对课堂展示是不合格的。

所以当前网页端采用的是“服务端预处理 + 浏览器直接展示”的方案。

### 8.1 当前公式渲染链路

回答生成后，不是直接把纯文本传给前端，而是走下面这条流程：

1. 模型输出 Markdown 和 LaTeX
2. 服务端用 `markdown-it-py` 把 Markdown 转成 HTML
3. 服务端用 `latex2mathml` 把 LaTeX 公式转成 MathML
4. 前端把 `answer_html` 直接插入页面

相关逻辑在 [stats_rag_web.py](stats_rag_web.py) 中的：

- `render_formula_html()`
- `render_answer_html()`
- `/api/ask` 返回的 `answer_html`

这样做有两个好处：

- 不依赖外部 CDN 的 MathJax
- 课堂展示更稳，断网时也不容易失效

### 8.2 为什么来源区公式不强求完美

来源区展示的是从 PDF 中抽取出的文本片段，而 PDF 文本抽取对复杂公式天然不够可靠。

所以当前策略是：

- 回答区尽量把模型生成的公式渲染好
- 来源区保留文本片段，强调可追溯
- 如果要核对最准确的公式排版，直接打开原 PDF

这是一种工程上的权衡：优先保证可解释性和展示稳定性。

## 9. 这个项目里每个组件的职责

为了方便课堂汇报，可以把每个组件总结成下面这样：

- `pypdf`：从讲义 PDF 中提取文本
- 文本清洗逻辑：去掉页眉页脚、页码和重复噪声
- chunk 切分：把长文档变成适合检索的语义片段
- Embedding 模型：把问题和知识片段映射到向量空间
- Chroma：存储向量并完成相似度召回
- Rerank 模型：从召回结果中选出更适合回答的问题证据
- Generation 模型：根据证据组织中文答案
- Web 服务：把问答过程做成可展示的交互界面
- MathML 渲染：让统计公式在网页中正常显示

## 10. 为什么这里显式禁用了代理继承

项目中 `httpx.Client(trust_env=False)` 和 `requests.Session().trust_env = False` 不是多余代码，而是为了解决真实部署中的一个问题：

- 如果系统开启了代理、TUN 或全局网络接管
- Python 请求库可能自动继承这些环境代理
- 结果就是访问模型 API 时出现连接错误、超时或者走错网络链路

因此当前项目强制让 Embedding、Rerank、Generation 走显式配置的 API 地址，不继承系统代理环境。

这属于很典型的“能运行”和“稳定运行”之间的工程差别。

## 11. 当前参数为什么这样定

这套参数是为了在“课堂展示效果”和“成本/速度”之间做平衡：

- `CHUNK_TARGET_CHARS=550`
  目的：让 chunk 既有足够语义，又不至于太大

- `CHUNK_OVERLAP_PAGES=1`
  目的：减少页与页之间切断概念的风险

- `RETRIEVAL_TOP_K=12`
  目的：给 rerank 足够候选，不要一开始就召回过窄

- `RERANK_TOP_N=5`
  目的：把上下文压缩到生成模型可控的范围

- `EMBEDDING_DIMENSIONS=1024`
  目的：与当前 embedding 模型接口保持一致

对 RTX 4050 这类设备来说，由于推理主要走远程 API，本地压力主要在：

- PDF 处理
- 构建向量库
- 本地网页服务

因此这套体量对你的机器是合适的，适合做课程项目第一版。

## 12. 当前方案的优点与局限

### 优点

- 知识来源明确，适合课堂展示
- 中文统计学领域聚焦，问题命中率较高
- 检索、重排、生成三段式结构清晰
- 可以展示来源页码，便于解释“答案从哪里来”
- 网页端支持公式显示，展示效果较好

### 局限

- PDF 文本抽取对复杂公式和表格不够理想
- 当前知识库规模不大，更偏“课程讲义问答”
- 还没有做多路召回、混合检索和更细粒度 chunk 策略
- 回答质量仍然依赖知识库覆盖度

## 13. 后续可以继续优化的方向

如果之后你想继续迭代，可以按下面这些方向升级：

- 扩充知识库：加入更多高校讲义、教材摘要或统计局公开资料
- 优化切分：按标题、段落或主题切 chunk，而不是只按页聚合
- 混合检索：结合 BM25 和向量检索
- 查询改写：针对统计学术语做 query expansion
- 多轮对话：保留历史提问上下文
- 更强展示：加入引用高亮、知识来源卡片、统计公式专栏

## 14. 课堂展示时可以怎么讲

如果你要在课堂上用 3 到 5 分钟介绍这套系统，可以按这个顺序说：

1. 我做的是一个面向统计学课程讲义的中文 RAG 系统
2. 知识库来自清华大学公开统计讲义
3. 系统先把 PDF 转成文本并切成 chunk，再做向量化存入 Chroma
4. 提问时先做向量召回，再用 rerank 精排
5. 最后把高相关证据交给大模型生成回答
6. 网页端还能把 Markdown 和公式渲染出来，便于展示
7. 因此它比普通聊天模型更可控，也更方便解释答案依据

如果老师问“为什么不是直接让大模型回答”，你可以回答：

因为统计学问题需要尽量依据课程讲义，而 RAG 可以让回答建立在可检索、可引用的资料上，减少无依据生成。

如果老师问“为什么还要 rerank”，你可以回答：

因为向量召回只能先找一批大致相关的内容，而 rerank 能进一步判断哪几段最适合真正回答当前问题。

## 16. 一句话总结

这套项目的本质不是“训练了一个统计学模型”，而是“把统计学讲义做成了一个可检索、可引用、可展示的问答系统”。

这正是 RAG 在课程项目里最有价值的地方。
