# 统计学 RAG 指标体系与调优导读

本文档面向第一次接触本项目评测工作的组员。

目标不是只告诉你“最后分数是多少”，而是先把下面三件事讲清楚：

- 我们为什么要给 RAG 做评测
- 每个指标到底在衡量什么
- 看到某种指标变化时，应该把问题归因到哪一层

建议把本文档作为阅读实验结果前的前置知识。

## 一、为什么 RAG 不能只看一个总分

RAG 不是一个单独模型，而是一条链路：

1. 用户提问
2. 系统先去知识库里检索相关内容
3. 再把检索到的内容交给大模型生成答案
4. 如果问题超出知识库范围，还要决定是否拒答

所以，RAG 的失败原因至少有三种：

- 没找到对的证据
- 找到了证据，但答案没答好
- 明明没有证据，却硬答了

这就是为什么我们不能只看一个“准确率”。

如果只看一个总分，你会不知道问题到底出在：

- 检索层
- 生成层
- 拒答层

因此，本项目采用的是分层指标体系。

## 二、我们这套指标体系在评什么

我们把指标分成三层：

- 检索层
- 生成层
- 拒答层

### 1. 检索层：系统能不能先把对的证据找出来

检索层回答的是一个最基础的问题：

> 真正相关的讲义内容，能不能出现在前面？

如果这一步做不好，后面的回答模型再强，也只能在错误上下文上作答。

本项目采用三个检索指标：

- `Hit@3`
- `MRR@5`
- `Evidence Recall@5`

#### `Hit@3`

它看的是：

> 前 3 个检索结果里，是否至少出现了 1 个标准证据。

直观理解：

- 高：说明正确证据经常能排进前 3
- 低：说明模型经常“找偏了”

为什么它重要：

- 课堂展示里，大模型通常只会基于前几条上下文作答
- 所以前 3 条能不能命中，是非常关键的现实指标

#### `MRR@5`

它看的是：

> 正确证据第一次出现得有多靠前。

举例：

- 第 1 条就命中，得分是 `1.0`
- 第 2 条才命中，得分是 `0.5`
- 第 5 条才命中，得分是 `0.2`

直观理解：

- `Hit@3` 看“有没有命中”
- `MRR@5` 看“命中得早不早”

它很适合用来观察 rerank 是否真的有效。

#### `Evidence Recall@5`

它看的是：

> 如果一道题需要 1 到 2 段关键证据，前 5 条里覆盖了多少。

直观理解：

- 高：说明系统不只是碰巧找到一点相关内容，而是把关键证据找得比较全
- 低：说明答案可能只拿到一半上下文

### 2. 生成层：系统拿到证据后，能不能答对、答完整、答得有依据

检索对了，不代表回答就一定好。

生成层回答的是：

> 在给定证据的前提下，大模型有没有正确、完整、且基于证据地回答问题。

本项目采用三个生成指标：

- `Answer Accuracy`
- `Key Point Coverage`
- `Faithfulness`

#### `Answer Accuracy`

它看的是：

> 这道题最终到底答得对不对。

在本项目里，由 judge model 按 `0` 到 `4` 分打分，再归一化到 `0` 到 `1`。

直观理解：

- 高：说明最终答案基本正确
- 低：说明要么漏关键点，要么理解错了，要么离题了

它是最接近“用户真实感受”的指标。

#### `Key Point Coverage`

它看的是：

> 标准答案中的关键点，模型到底覆盖了多少。

例如一道题标准有 3 个关键点：

- 只答到 2 个，覆盖率就是 `2/3`
- 全答到，覆盖率就是 `1.0`

直观理解：

- `Answer Accuracy` 看整体对不对
- `Key Point Coverage` 看具体漏了什么

这个指标非常适合调 prompt，因为很多时候答案“方向对”，但“不够全”。

#### `Faithfulness`

它看的是：

> 模型答案中的关键结论，是否都能从检索证据中得到支持。

这是 RAG 和普通大模型问答最大的区别。

直观理解：

- 高：说明答案真正是“基于知识库在答”
- 低：说明答案里出现了“模型自己补出来”的内容

这个指标的优先级非常高，因为课堂展示里最怕的不是“没答全”，而是“讲义里没有，它却编了”。

### 3. 拒答层：越界问题时系统能不能稳住

不是所有问题都应该回答。

如果当前知识库没有覆盖某个概念，比如 `SVM`、`Cox 回归`、`LSTM`、`Transformer`，那么一个合格的 RAG 系统应该拒答，而不是凭模型常识乱讲。

本项目采用两个拒答指标：

- `Refusal F1`
- `Over-Refusal Rate`

#### `Refusal F1`

它看的是：

> 该拒答的问题，有没有拒；不该拒答的问题，有没有误拒。

它综合考虑了：

- 拒答是否准确
- 拒答是否完整
- 有没有漏掉该拒答的问题

如果这个分数高，说明系统的安全边界比较稳。

#### `Over-Refusal Rate`

它看的是：

> 明明能回答的问题，系统却保守地拒答了多少。

直观理解：

- 高：系统太怂，不好用
- 低：系统在该答时愿意答，在该拒时才拒

## 三、怎么通过指标判断问题出在哪一层

这是最关键的一部分。

我们做调优时，不是看到分数变了就说“好”或“不好”，而是要会读指标组合。

### 情况 1：`Hit@3` 低，但 `Faithfulness` 很高

通常意味着：

- 检索没找准
- 但模型至少没有乱编

这类问题优先从检索层调：

- embedding query instruction
- rerank instruction
- chunk 划分
- top_k / rerank_top_n

### 情况 2：`Hit@3` 不低，但 `Answer Accuracy` 和 `Key Point Coverage` 低

通常意味着：

- 证据其实找到了
- 但模型没把关键点讲全

这类问题优先从生成 prompt 调：

- 要不要强调“覆盖所有关键点”
- 要不要限制答得太散
- 对不同题型要不要单独约束回答方式

### 情况 3：`Answer Accuracy` 高，但 `Faithfulness` 低

通常意味着：

- 答案看起来对
- 但不是根据讲义在答，而是靠模型常识在补

这类问题最危险。

要优先做：

- 更严格的“只能依据检索内容回答”约束
- 证据不足时直接拒答
- 对越界术语做强制保护

### 情况 4：`Refusal F1` 低

通常意味着：

- 要么不该答时答了
- 要么该答时却拒了

这类问题优先看：

- 拒答规则是否明确
- 系统 prompt 是否对“证据不足时怎么办”讲得足够清楚
- 是否对明显越界术语做了显式保护

## 四、为什么我们这次调优是按这个顺序来的

本项目的调优顺序不是随便排的，而是按照风险优先级来的。

### 第一步：先把“乱答”压住

最开始的基线版本，虽然 `Answer Accuracy` 很高，但 `Refusal F1` 不够高，`Faithfulness` 也不是满分。

这说明它的问题不是“不会说”，而是“太敢说”。

所以第一轮优化优先做的是：

- 强化拒答
- 强化忠实度

这是因为在课堂展示里：

- 答得稍微短一点，还能接受
- 但明明讲义里没有，它却一本正经地讲错，这个风险更大

### 第二步：再把“过于保守”拉回来

第一轮优化之后，`Faithfulness` 和 `Refusal F1` 都变成了 `1.0`，但 `Answer Accuracy` 和 `Key Point Coverage` 掉得比较明显。

这说明系统变成了另一种极端：

- 不乱讲了
- 但也不敢讲全了

所以第二轮优化的核心是：

- 不碰检索主线
- 只调生成 prompt
- 明确要求“证据充分时把关键点讲全”

### 第三步：再尝试检索增强，但只保留真正有效的部分

第三轮里做过一次更激进的检索实验，比如：

- 多 query 召回
- 术语扩展
- 去重重排

但完整评测表明，这套方案没有带来稳定收益，反而会引入一些噪声。

这一步很重要，因为它说明：

> 不是所有“看起来更聪明”的检索策略都会让系统变好。

最后保留下来的，只是对最终结果稳定有帮助的那部分改动。

## 五、如何正确阅读我们最后的实验结果

当前最终版本是：

- `runs/20260404_192700/summary.json`

它的关键指标为：

- `Hit@3 = 0.7500`
- `MRR@5 = 0.6641`
- `Evidence Recall@5 = 0.8281`
- `Answer Accuracy = 0.9062`
- `Key Point Coverage = 0.9688`
- `Faithfulness = 1.0000`
- `Refusal F1 = 1.0000`
- `Over-Refusal Rate = 0.0000`

### 应该怎样理解这组结果

可以按下面方式理解：

- 检索层：已经达到可展示水平，但还不是最强
- 生成层：已经很强，尤其是正确率和关键点覆盖率
- 拒答层：已经非常稳
- 忠实度：已经非常稳

换句话说：

- 当前系统最大的优点，不是“特别会说”，而是“说得对、说得有依据、超纲时能稳住”
- 当前最大的剩余提升空间，不在拒答，也不在忠实度，而在检索排序

## 六、为什么我们说当前表现已经优秀

如果只看一个指标，容易得出偏差结论。

但如果从课堂展示目标出发，当前版本确实已经可以评价为优秀：

- `Answer Accuracy` 超过 `0.90`
- `Key Point Coverage` 接近 `0.97`
- `Faithfulness = 1.0`
- `Refusal F1 = 1.0`

这意味着：

- 系统答大多数题是靠谱的
- 答案基本建立在知识库证据上
- 对知识库外的问题不会乱说

不过，我们也不会把它说成“已经顶级”，因为：

- `Hit@3` 还只是 `0.75`
- `assumption` 和 `method_selection` 两类题的检索还不够强

所以，最准确的评价是：

- 这是一个优秀的课堂展示版本
- 不是一个已经完全打磨到头的检索系统

## 七、你们看实验时应该抓住什么重点

当你和组员一起看实验记录时，建议优先抓住三件事：

### 1. 先看主指标，而不是先看总表是否好看

先看：

- `Hit@3`
- `Answer Accuracy`
- `Faithfulness`
- `Refusal F1`

因为这四个指标分别代表：

- 找没找到证据
- 最终答没答对
- 是不是基于证据在答
- 面对越界问题能不能稳住

### 2. 再看指标之间有没有“此消彼长”

RAG 调优经常不是单向提升，而是平衡问题。

例如：

- 拒答更稳了，回答可能变短
- 回答更完整了，可能又会开始多补一点无关内容

所以，不要只问“这个指标升了没”，要问：

> 它是用什么代价换来的？

### 3. 最后再看逐题误差分析

逐题分析的意义不是“挑错”，而是帮助你判断下一轮该调哪里。

比如：

- 如果很多 definition 题都漏“重复抽样”这种关键词，说明 definition prompt 还不够精准
- 如果很多 assumption 题命中率低，说明检索层还要继续针对假设类问题做优化

建议小组成员按下面顺序读：

1. 本文档 `指标体系与调优导读.md`

不会出现“先看到一堆分数，但不知道这些分数在说什么”的问题。

## 九、最终一句话结论

如果你要把这套评测体系讲给组员听，可以先记住下面这句话：

> 我们不是只在评“答案像不像对”，而是在同时评四件事：有没有找到证据、有没有答对、是不是基于证据在答、以及面对知识库外问题能不能稳住。

只要把这句话讲明白，后面的实验结果就比较容易看懂了。
